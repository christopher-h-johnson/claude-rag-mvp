# S3 CORS Configuration Fix

## Problem
When uploading files to S3 using presigned URLs, the browser was blocking the request with a CORS error:
```
Access to XMLHttpRequest at 'https://...s3.amazonaws.com/...' from origin 'http://localhost:5173' 
has been blocked by CORS policy: Response to preflight request doesn't pass access control check: 
No 'Access-Control-Allow-Origin' header is present on the requested resource.
```

## Root Cause
The S3 bucket (`dev-chatbot-documents-*`) did not have CORS configuration, so it was rejecting cross-origin requests from the frontend application.

## Solution
Added CORS configuration to the S3 bucket in Terraform to allow uploads from the frontend origin.

### Changes Made

1. **Added CORS variable to storage module** (`terraform/modules/storage/variables.tf`):
   ```hcl
   variable "cors_origin" {
     description = "CORS origin for S3 bucket (e.g., http://localhost:5173 for development)"
     type        = string
     default     = "http://localhost:5173"
   }
   ```

2. **Added CORS configuration resource** (`terraform/modules/storage/main.tf`):
   ```hcl
   resource "aws_s3_bucket_cors_configuration" "documents" {
     bucket = aws_s3_bucket.documents.id

     cors_rule {
       allowed_headers = ["*"]
       allowed_methods = ["PUT", "POST", "GET", "HEAD"]
       allowed_origins = [var.cors_origin]
       expose_headers  = ["ETag", "x-amz-server-side-encryption", "x-amz-request-id", "x-amz-id-2"]
       max_age_seconds = 3000
     }
   }
   ```

3. **Passed cors_origin to storage module** (`terraform/main.tf`):
   ```hcl
   module "storage" {
     source = "./modules/storage"
     # ... other variables
     cors_origin = var.cors_origin
   }
   ```

## CORS Configuration Details

- **Allowed Headers**: `*` (all headers)
- **Allowed Methods**: `PUT`, `POST`, `GET`, `HEAD`
- **Allowed Origins**: Configured via `cors_origin` variable (default: `http://localhost:5173`)
- **Exposed Headers**: `ETag`, `x-amz-server-side-encryption`, `x-amz-request-id`, `x-amz-id-2`
- **Max Age**: 3000 seconds (50 minutes) - browser caches preflight response

## Deployment

Apply the Terraform changes:
```bash
cd terraform
terraform apply
```

The CORS configuration will be applied to the S3 bucket, allowing the frontend to upload files directly to S3 using presigned URLs.

## Production Configuration

For production deployment, update the `cors_origin` variable in `terraform.tfvars`:

```hcl
cors_origin = "https://your-production-domain.com"
```

Or if you need multiple origins, you can modify the CORS rule to accept a list:

```hcl
# In terraform/modules/storage/variables.tf
variable "cors_origins" {
  description = "List of CORS origins for S3 bucket"
  type        = list(string)
  default     = ["http://localhost:5173"]
}

# In terraform/modules/storage/main.tf
cors_rule {
  allowed_origins = var.cors_origins
  # ... rest of config
}
```

## Testing

After applying the changes:

1. Open the frontend at `http://localhost:5173`
2. Login to the application
3. Try uploading a PDF document
4. The upload should succeed without CORS errors
5. Check the browser console - no CORS errors should appear
6. Verify the file appears in S3 under `uploads/{documentId}/`

## Security Notes

- The CORS configuration only allows specific methods (PUT, POST, GET, HEAD)
- The bucket still has public access blocked
- Uploads require presigned URLs generated by the authenticated upload Lambda
- The presigned URLs expire after 15 minutes
- All uploads are encrypted at rest using KMS

## Troubleshooting

If CORS errors persist:

1. **Verify Terraform applied successfully**:
   ```bash
   aws s3api get-bucket-cors --bucket dev-chatbot-documents-177981160483
   ```

2. **Check the CORS origin matches**:
   - Frontend origin: `http://localhost:5173`
   - CORS allowed origin: Should match exactly (including protocol and port)

3. **Clear browser cache**:
   - Browsers cache CORS preflight responses
   - Hard refresh (Ctrl+Shift+R) or clear cache

4. **Check presigned URL**:
   - Ensure the presigned URL is generated correctly
   - Verify it includes all required query parameters
   - Check it hasn't expired (15-minute TTL)

5. **Verify S3 bucket policy**:
   - Ensure the bucket policy doesn't block the upload
   - Check that TLS 1.2+ is being used
